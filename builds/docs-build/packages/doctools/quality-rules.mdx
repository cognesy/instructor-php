# Quality Rules

Doctools quality checks are rule-driven.

Rules are loaded from:
1. Profile baseline (`packages/doctools/resources/config/quality/profiles/<profile>.yaml`)
2. Package-local docs rules (`<docs-root>/.qa/rules.yaml`)
3. Explicit CLI files (`--rules`), applied last

Later sources override earlier ones when rule IDs are the same.

Bundled profiles currently include:

- `instructor`
- `http-client`
- `polyglot`
- `none`

## Coverage Model

Docs QA coverage expands in two layers:

1. A shared package profile for stable, package-wide anti-drift rules
2. A package-local `.qa/rules.yaml` file for small, fast package-specific overrides

Use both when a package has public API drift risks and also a few docs-local conventions.

## When to Add a Shared Profile

Add a profile under `packages/doctools/resources/config/quality/profiles/` when all of these are true:

- the package has its own public API surface
- the same drift patterns are likely to appear across many docs pages
- the rules are stable enough to become part of the package baseline
- the package should be runnable through `--profile <package>`

Examples:

- removed factory methods
- renamed config keys
- deprecated facade entrypoints
- removed streaming or response APIs

## When Local `.qa` Rules Are Enough

Keep rules only in `<package>/docs/.qa/rules.yaml` when they are:

- temporary cleanup guards
- specific to one docs tree
- too experimental for a shared baseline
- intended to override or narrow the shared profile

If a local rule keeps proving useful across the whole package, promote it into a shared profile.

## Rule File Format

```yaml
version: 1
rules:
  - id: no_from_config
    engine: regex
    scope: markdown
    pattern: '/\bStructuredOutput::fromConfig\s*\(/'
    message: 'Use StructuredOutputRuntime::fromConfig(...) or StructuredOutput::withConfig(...).'

  - id: no_request_call
    engine: ast-grep
    scope: php-snippet
    language: php
    pattern: '$OBJ->request($$$ARGS)'
    message: 'Avoid direct request() calls in examples.'
# @doctest id="5e03"
```

### Required fields

- `id`
- `engine`: `regex` or `ast-grep`
- `scope`: `markdown` or `php-snippet`
- `pattern`
- `message`

For `ast-grep` rules, `language` is required.

## CLI Usage

```bash
php bin/instructor-docs qa \
  --source-dir packages/polyglot/docs \
  --profile polyglot \
  --rules packages/polyglot/docs/.qa/rules.yaml
# @doctest id="b2aa"
```

JSON output for CI:

```bash
php bin/instructor-docs qa \
  --source-dir packages/polyglot/docs \
  --profile polyglot \
  --format json
# @doctest id="e3fd"
```

If `ast-grep` rules are configured, missing binary fails by default.  
Use `--no-strict` to continue without ast-grep checks.

## Expanding QA to a New Package

### Minimal rollout

Use this when a package only needs a few checks:

1. Add `<package>/docs/.qa/rules.yaml`.
2. Start with high-signal regex rules.
3. Run `php bin/instructor-docs qa --source-dir packages/<package>/docs --profile none`.
4. Fix drift in the docs before adding more rules.

### First-class rollout

Use this when the package needs a reusable baseline:

1. Add `packages/doctools/resources/config/quality/profiles/<package>.yaml`.
2. Add `<package>/docs/.qa/rules.yaml` for package-local exceptions or additions.
3. Update CLI/help text so the new profile is discoverable.
4. Add feature coverage proving the profile catches a representative stale API pattern.
5. Run `php bin/instructor-docs qa --source-dir packages/<package>/docs --profile <package>`.

## Files to Update for a New First-Class Profile

When introducing a new bundled profile, update these places together:

- `packages/doctools/resources/config/quality/profiles/<package>.yaml`
- `packages/<package>/docs/.qa/rules.yaml`
- `packages/doctools/src/Quality/Commands/RunDocsQualityCommand.php`
- `packages/doctools/docs/quality-rules.md`
- `packages/doctools/CHEATSHEET.md`
- doctools feature tests for `qa`

That keeps the profile runnable, documented, and covered by tests.

## How to Derive Good Rules

Start from real API drift, not abstract style preferences.

A practical process:

1. Compare the package docs and cheatsheet to the current public API.
2. List stale calls, stale config keys, and removed concepts.
3. Keep only rules that are likely to recur.
4. Prefer one rule per concrete drift pattern.
5. Write messages that tell the writer what to use instead.

Good rule targets:

- removed methods such as `withOutputMode(...)`
- removed classes or enums such as `OutputMode`
- renamed config keys such as `providerType` â†’ `driver`
- removed response accessors such as `stream()->responses()`

Avoid low-signal rules such as general writing preferences or package-internal implementation details that docs readers should never see anyway.

## Rule Design Guidance

Prefer:

- regex rules for exact token or method anti-drift checks in markdown
- ast-grep rules for PHP snippet structure that regex would handle poorly

Keep rules:

- narrow
- explicit
- easy to explain
- stable across releases

Rule IDs should stay stable. If a package-local rule intentionally overrides a profile rule, reuse the same rule ID.

## Rollout Checklist

1. Audit the package docs against the current public API.
2. Decide whether the package needs only local rules or a first-class profile.
3. Add high-signal anti-drift rules first.
4. Run `php bin/instructor-docs qa --source-dir <package>/docs --profile <package-profile-or-none>`.
5. Add feature tests when a new bundled profile is introduced.
6. Add `--format json` in CI for machine-readable reports.
7. Keep rule IDs stable to support predictable overrides.
